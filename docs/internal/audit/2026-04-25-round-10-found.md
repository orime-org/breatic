### BUG-192:AI 聊天前端是 setTimeout(3000) mock,核心产品功能完全不通

- **严重度**:🔴 P0 HIGH(产品功能完整性)
- **位置**:`packages/web/src/apps/project/components/agent/AiChatRecordPanel.tsx:309-330`
- **代码原貌**:

  ```tsx
  const handleSend = useCallback((content: string) => {
    const userMsg: MessageItem = { id: nanoid(), type: 'user', senderName: 'You', content };
    const assistantMsg: MessageItem = { id: nanoid(), type: 'assistant', senderName: 'Assistant', content: '', loading: true };
    setMessageList((prev) => [...prev, userMsg, assistantMsg]);
    const assistantId = assistantMsg.id;
    window.setTimeout(() => {
      setMessageList((prev) =>
        prev.map((m) => (m.id === assistantId ? { ...m, loading: false, content: assistantReplyAfterLoading } : m)),
      );
    }, 3000);
  }, []);
  ```

- **问题**:`handleSend` 完全 mock —— 无任何 API / SSE 调用。后端 Agent + SubAgent + 三层记忆 + Turn 压缩等所有 chat 链路代码**前端从未触发**

- **影响**:
  1. 核心产品功能(AI 协作)在 UI 上不工作
  2. 此前 R3~R8 关于 chat 链路的 bug 审计(BUG-079 / BUG-112 / BUG-117 / BUG-185 等)基于"前端在调后端"假设,部分 active bug 实际**不会触发**(因为代码路径 dead)
  3. **未知**:这是 regression 还是从未实现? **dev 修此 bug 时必须在 PR description 回答**

- **测试规约**:
  - **Input**:在 web UI 打开任意 project,在 AI Chat 面板输入文本 + 点 send 按钮
  - **Procedure**:开浏览器 DevTools Network 面板观察
  - **Expected**:有 `POST /api/v1/chat/message` SSE 请求发出
  - **Current**:Network 零请求,3 秒后界面显示 mock reply
  - **Pass**:真实 SSE 请求触发 + 响应流式返回 + 错误正确反馈
  - **Fail**(当前):零请求 + mock 返回 + 后端 Agent 全空跑

- **修复方向**:dev 需要 (a) 调通 `apis/chat.ts` 的 `sendMessage()` 真接口 (b) 处理 SSE 事件流 (c) abort 处理 (依赖 BUG-193 / BUG-194 一并修)。**严格说不算单个文件 fix,是补全功能**

### BUG-193:`sse.ts` 的 `onerror` 不调用用户传入的 callback,所有 SSE 错误对 caller 静默失败

- **严重度**:🟠 P1 MED(错误处理 / UI 可靠性)
- **位置**:`packages/web/src/utils/sse.ts:90-92`
- **代码原貌**:

  ```ts
  onerror: (err) => {
    throw err;
  },
  ```

- **问题**:`fetchEventSource` 的 `onerror` 中只 throw 不 retry,但**没有调用 `config.onerror?.(err)`**。所有调 `sse({ ..., onerror: myHandler })` 的代码,`myHandler` 永远不被触发。caller UI 看到的是"流静默断"

- **测试规约**:
  - **Input**:调用 `sse({ url: '/api/v1/non-existent', onerror: (err) => { spy(err) } })`,触发 404
  - **Expected**:`spy` 被调用 1 次,参数是 Error
  - **Current**:`spy` 0 调用,fetchEventSource 内部 throw 后 sse helper 返回 promise reject
  - **Pass**:用户 onerror 收到通知
  - **Fail**(当前):静默
  - **边界**:401 / 402 / 5xx 也应触发 onerror;abort signal 触发的 close 不算 error

- **修复方向**:在 `throw err` 前先 `config.onerror?.(err)`,或改成 `if (config.onerror) config.onerror(err); throw err;`

### BUG-194:`sse.ts` 402 错误抛通用 `Failed to open stream` —— 用户看不到"积分不足"+ 无购买 CTA

- **严重度**:🟠 P1 MED(UX + 产品策略 #2)
- **位置**:`packages/web/src/utils/sse.ts:71-80`
- **问题**:

  ```ts
  if (response.ok) { ... return; }
  if (response.status === 401) { removeToken(); navigate('/login'); }
  throw new Error(`Failed to open stream: ${response.status} ${response.statusText}`);
  ```

  402(Payment Required,积分不足)走最末通用 throw 路径,UI 拿到的是 `Failed to open stream: 402 Payment Required` 这种神秘错误。**没有"购买积分"专属处理**

### BUG-195:`request.ts` 网络错误时 `decrementLoading()` 不调,全局 spinner 永转

- **严重度**:🟠 P1 MED(UX,看似死机)
- **位置**:`packages/web/src/utils/request.ts:50-53`
- **代码原貌**:

  ```ts
  const isNetworkError = !error.response && (error.code === 'ECONNABORTED' || error.code === 'ERR_NETWORK' || error.message === 'Network Error');
  if (customConfig?.needGlobalLoading === true) {
    if (!isNetworkError) {
      store.dispatch(decrementLoading());
    }
  }
  ```

- **问题**:request 入口先 `incrementLoading()`(行 32),失败时若是网络错误(`!error.response`),`decrementLoading()` **永远不调** → loading counter 累加且永不清零,全局 spinner 永转

- **测试规约**:
  - **Input**:断开网络,触发 1 个 `needGlobalLoading: true` 请求
  - **Expected**:请求失败后 `store.getState().loading.counter === 0`(spinner 消失)
  - **Current**:counter === 1,spinner 永转
  - **Pass**:loading counter 在请求失败后回零
  - **Fail**(当前):counter 持续累加
  - **边界**:测试时 spinner 不应"卡顿"或"闪一下消失再亮起",必须真实回零

- **修复方向**:删除 `if (!isNetworkError)` 条件即可(网络错误也应 decrementLoading)。当前 if 的本意可能是"网络错误期间 spinner 留着等用户重试"——但这违反 finally 模式应该一律 decrement

### BUG-203:`request.ts:45` 不解响应信封,每个 caller 必须手动 `if (!res.success)`

- **严重度**:🟡 P2 LOW(可维护性 / DRY)
- **位置**:`packages/web/src/utils/request.ts:45`
- **问题**:`return response.data` 直接返回,不检查 `success: false`。后端返回 `{ success: false, error: 'foo' }` 状态 200 时,前端拿到的不是 reject,需要每个 API caller 手动 `if (!res.success) throw ...`

- **影响**:DRY 违反 + 易错(漏检的 caller 把 error 当 success 处理)

- **测试规约**:
  - **Input**:后端返回 `{ success: false, data: null, error: 'foo' }` 状态 200
  - **Expected**:axios interceptor 把它当 reject
  - **Current**:caller 拿到 `{ success: false, ... }` 当 success 处理
  - **修复方向**:在 interceptor 加 `if (!response.data.success) throw new ApiError(response.data.error)`
